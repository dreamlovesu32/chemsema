### Summary

Today's main thread was turning the Windows-native development environment from May 5 into the beginning of a real Windows desktop product. The project is no longer merely a viewer that can run on Windows. It now has the early shape of a desktop architecture: a Tauri 2 shell, a Rust native document service, desktop filesystem capabilities, Windows clipboard integration, multi-document windows, native/custom desktop chrome, PDF/EMF preview export, and the boundaries needed for future Office/OLE integration.

At the same time, the viewer and Rust engine continued to clarify their responsibilities. The browser still uses WASM. The desktop app now probes the native Rust engine through Tauri commands, but the main editing loop remains on the synchronous WASM model for now. The UI talks through host abstractions such as `EngineHost`, `DesktopFileHost`, and `ColorHost`, so Web, Desktop, and future Office layers do not fork chemistry behavior.

Engine correctness and CDXML fidelity also advanced. Overvalent oxygen label recognition was fixed: a neutral three-connection oxygen is invalid, while a positively charged three-connection oxygen remains valid. CDXML import/export was stabilized around double-bond placement, ACS/Default drawing options, wedge width, label anchors, centered labels, and numeric suffix labels. The last work of the day added selection resize handles, so the select tool can now scale selected objects in addition to moving, rotating, arranging, flipping, and recoloring them.

### Windows-Native Scripts And Desktop Architecture

The day started by moving active npm scripts from Bash to Node. `build:engine-wasm`, `dev:engine`, and `verify` no longer call `bash scripts/*.sh`; they now run:

```text
node scripts/build-engine-wasm.mjs
node scripts/dev-engine-wasm.mjs
node scripts/verify.mjs
```

The old Bash scripts were removed. `build-engine-wasm.mjs` calls `wasm-pack build` through Node `spawnSync`, then removes the `.gitignore` generated by wasm-pack under `viewer/engine` so runtime artifacts remain tracked. `dev-engine-wasm.mjs` uses Node `fs` and `crypto` to scan `Cargo.toml`, `Cargo.lock`, and engine `src/**/*.rs`, compute a SHA-256 hash, and rebuild WASM when inputs change. It no longer depends on Unix tools such as `find`, `sha256sum`, or `awk`. `verify.mjs` runs `cargo test`, builds WASM, checks `viewer/app.js`, and verifies that `viewer/engine` generated artifacts are synchronized.

The practical result is that native Windows PowerShell is now a first-class development path. Git Bash may still be useful, but the main npm workflow no longer requires Bash.

The new `docs/windows-desktop-office-architecture.zh-CN.md` records the long-term Windows desktop and Office integration architecture, and `docs/project-rules.zh-CN.md` now points to it. The direction is:

```text
crates/chemcore-engine
  The only chemistry core: document model, editing commands, CDXML, SVG, render primitives, hit testing.

crates/chemcore-desktop-service
  Desktop document service: native engine sessions, file I/O, recent files, export, future recovery.

apps/chemcore-desktop
  Tauri Windows desktop app: windows, commands/menus, dialogs, clipboard, file associations, drag/drop open.

apps/chemcore-office
  Future Office/OLE integration: embeddable and double-click-editable Chemcore objects with previews.

viewer/
  Shared Web/Desktop editing UI.
```

The architecture explicitly rejects a few paths: no temporary Electron app, no desktop-specific chemistry editing logic, no Office plugin directly parsing Chemcore JSON, no SVG-only paste path that postpones editable objects indefinitely, and no `.ccjz` API that is permanently locked to one gzip JSON file. `.ccjz` remains gzip JSON in the first phase, but it should be exposed through stable container APIs to leave room for previews, resources, thumbnails, and Office object payloads.

### Tauri Shell And Native Engine Boundary

`apps/chemcore-desktop/src-tauri` was added today. The Tauri app joined the root Cargo workspace and includes `Cargo.toml`, `build.rs`, `src/main.rs`, `src/lib.rs`, capabilities, Tauri configuration, and default application icons. `tauri.conf.json` points at the existing `viewer/`, uses `http://127.0.0.1:8767/viewer/` in dev, and creates a 1280x900 window with a 960x640 minimum size.

The npm desktop commands are:

```text
npm run desktop:dev
npm run desktop:build
npm run desktop:info
```

`scripts/desktop-tauri.mjs` invokes the local Tauri CLI from `apps/chemcore-desktop`. `scripts/desktop-dev-server.mjs` is a small Node static server for the repo, returning correct MIME types for JS, CSS, WASM, SVG, PNG, and related assets. If port 8767 is already in use, it keeps the process alive so Tauri dev can reuse the existing server.

The new `crates/chemcore-desktop-service` directly depends on `chemcore-engine`. It manages native engine sessions through `DesktopDocumentService`, with APIs for session create/free, JSON/CDXML loading, document JSON, state JSON, render list, render bounds, CDXML export, and SVG export. Tests cover session lifecycle and blank-session document/render JSON.

The Tauri backend stores `DesktopDocumentService` in state and exposes `desktop_engine_*` commands. The frontend added `viewer/engine_host.js`, replacing direct `new WasmEngine()` usage with `engineHost.createEngineSession()`. Web uses `WasmEngineHost`. Tauri currently uses `DesktopHybridEngineHost`: the editor UI still uses a WASM session, while startup also creates a native session, reads document JSON, render list, bounds, and SVG, then frees the session. This proves that the Tauri command path can call the native Rust engine without disturbing the existing editor loop.

That hybrid path is intentional. It starts the native desktop engine route without forcing the whole synchronous editor model to change in one step. Future migration should continue by implementing `TauriEngineHost`, not by scattering raw Tauri command calls through UI code.

### Desktop Files, Clipboard, Export, And Windows

The desktop service grew from an engine-session wrapper into a file service:

- `.ccjz` uses native Rust gzip read/write through `flate2`.
- `.ccjs`, `.cdxml`, and `.svg` use text read/write.
- Unknown save extensions default to `.ccjz`.
- Gzip magic bytes are recognized even if the extension is ambiguous.
- Text content that looks like CDXML is treated as CDXML.
- Recent files are persisted under the user data directory at `Chemcore/desktop/recent-files.json`, capped at 10 entries, with existence flags returned to the UI.

The Tauri backend added native file dialogs and file commands for opening, saving, choosing export paths, reading/writing paths, writing base64 export data, clearing recent files, and taking startup open paths. Desktop open/save no longer depends only on the browser File System Access API; it can work with real Windows file paths.

Clipboard support now uses Windows multi-format data. Desktop clipboard writes can include:

```text
Chemcore Clipboard Fragment
Chemcore Document JSON
ChemDraw Interchange Format
chemical/x-cdxml
image/svg+xml
SVG
Unicode text
```

Reads use the same formats, and the viewer chooses the paste priority. This lays the groundwork for better copying between Chemcore, Office, ChemDraw, browsers, and other apps.

Two preview export paths were added. PDF preview is built in the viewer: `viewer/export_preview.js` renders SVG to canvas, embeds a JPEG in a single-page PDF, and returns base64 for native file writing. EMF preview is built in the Tauri backend: it reads render list JSON and bounds JSON and uses Windows GDI to draw core render primitives such as lines, polygons, paths, rectangles, circles, ellipses, and text.

Window behavior moved toward a real desktop application. `viewer/index.html` added a custom `desktop-titlebar` with branding, document tabs, a new-tab button, and minimize/maximize/close controls. The Tauri backend added commands for setting the title, minimizing, toggling maximize, closing, starting window drag, reading maximized state, detaching a document window, and taking the detached-window payload. The frontend calls these through `DesktopFileHost`.

Multi-document tabs were a major UI addition. `viewer/app.js` now keeps a `documentTabs` list, with each tab storing its engine, file name, file path, zoom, title, and document state. New/Open/DnD/startup paths are tab-aware: opening a file creates a new tab, and closing the last tab creates a fresh Untitled tab. On desktop, dragging a tab below the titlebar can create a new Tauri window. The current document payload is staged in Tauri state and retrieved by the new window by label.

Native menu command wiring was also built for File/Edit/View operations, including recent files, CDXML/SVG/PDF/EMF export, undo/redo/cut/copy/paste/delete, and zoom. Later in the day `USE_NATIVE_MENU` was set to `false`, reflecting the current preference for custom desktop chrome and the HTML command system, avoiding split command sources between native menus and the focused WebView.

### Color Tools And Toolbar Experience

Color tooling converged in two steps. First, "which colors are used by the document" moved from frontend scanning into the Rust engine. `Engine::document_colors()` traverses the `ChemcoreDocument` JSON representation and collects string values under keys containing `color`, `fill`, `stroke`, or `background`. It supports `#rrggbb`, `#rgb`, and `rgb(r, g, b)`, skips `none` and empty strings, and returns a stable de-duplicated list. WASM and the desktop service expose this through `documentColorsJson()` / `document_colors_json()`, and the viewer's `currentDocumentColors()` asks the engine first.

Second, color choosing was centralized behind `ColorHost`. A Windows `ChooseColorW` path was briefly introduced, then removed in favor of a Chemcore-owned color dialog shared by Web and Desktop. The current `viewer/color_host.js` provides:

- Basic color palette.
- Custom color palette.
- HSV spectrum.
- Brightness slider.
- RGB / HSV / Hex inputs.
- Add to custom colors.
- OK/Cancel, Escape, and backdrop close.

This keeps the color picking experience consistent across Web and Desktop, without depending on Windows common dialog behavior.

Toolbar code and density were also cleaned up. `viewer/toolbar.js` moved many manually embedded SVG strings into generated icon helpers for bonds, arrows, shapes, brackets, rings, text formatting, and distribute controls. `viewer/styles.css` gained CSS variables for desktop/browser density: topbar height, secondary toolbar height, tool rail width, button sizes, icon sizes, stroke widths, and color panel size. Browser shell uses a more compact scale; desktop shell keeps proportions closer to a professional drawing application.

The select tool gained a selection color picker. The Rust engine's `apply_color_to_selection()` can recolor selected text objects, arrows, shapes, molecule objects, bonds, node labels, and label runs. The toolbar command modifies the document through the engine instead of only changing viewer-side state.

### Chemistry Semantics And CDXML Fidelity

The clearest chemistry fix today was overvalent oxygen recognition. Element/hydrogen label validation in `engine/text_edit/labels.rs` now receives the full fragment, sums bond orders connected to the node, and calls the implicit-hydrogen / typical-valence rules to decide whether the element label is chemically valid. Loading JSON or CDXML refreshes valence recognition for existing node labels. Tests verify:

- A neutral three-connection oxygen label `O` produces `labelRecognition.status = invalid` and a red invalid box.
- A three-connection oxygen with `+1` charge remains valid.

CDXML drawing options were stabilized. Imported CDXML bond length, line width, bold width, hash spacing, and bond spacing are preserved, but imported metrics are no longer treated as the currently selected UI preset. `document_style_preset` remains Default unless the user explicitly applies a preset. The frontend replaced the old style preset `<select>` with a Style button and menu, and applying a preset now shows a confirmation because it rescales the drawing and updates existing bond, label, and graphic metrics.

Double-bond import also improved. CDXML double bonds with explicit `DoublePosition=Left/Right/Center` become frozen placements. Double bonds without explicit positions but with imported line styles or weights also stay center/frozen, preventing dashed-solid style bonds from being auto-shifted. Only plain unspecified double bonds call the engine's `automatic_double_bond_placement_for_segment()`. Tests cover benzene inward-side placement, unspecified alkene placement, right-side double-bond rendering, and BondSpacing-derived spacing.

Wedge width now derives from `BoldWidth * 1.5`, with a default stroke-width floor, instead of relying on a fixed special case. ACS/Default label clip margin is also derived from imported bold width, avoiding a simplistic "ACS is just Default scaled" model.

Bond-contact rendering also had an important refactor, especially for hashed wedges and hash bonds. The old behavior mixed a few competing ideas: sometimes the hash/hashed-wedge mother polygon stayed fixed while neighboring bonds retreated; sometimes special center-double-outer-line intersection logic tried to compute the retreat. Today's implementation converges on clearer rules:

- The main bond contact kernel treats hash bonds and hashed wedges as hash contact obstacles, so they do not participate in the ordinary main-contact ring/kernel patch calculation.
- Hash bonds and hashed wedges actively retreat at unlabeled connected endpoints using the shared `hash_contact_retreat_distance_for_bond()`. That distance combines black segment length and target gap, scaled by stroke width.
- If the connected endpoint has a visible label, label clipping remains the controlling rule and no extra hash-contact retreat is added, avoiding over-shortened hash geometry near labels.
- Hashed wedge outline geometry and the internal knockout/hash pattern are computed separately: the outline retreats at connected endpoints, while the pattern still uses the original wedge span, preserving the visual black/white rhythm.
- The older large center-double-outer-line retreat/intersection block in `render/bond_geometry.rs` was removed in favor of simpler checks around whether the endpoint has another bond, whether it has a label, and whether the bond is a hash obstacle.
- Main contact patches now preserve bond stroke. If all bonds at a bridge share the same color, the bridge patch uses that shared color; otherwise individual patches keep each bond's stroke, so colored CDXML bonds are not overwritten by default black at joins.

Solid wedge and ordinary main-bond contacts were strengthened at the same time. Three-way and four-way main contacts now emit valid contact patches. A solid wedge entering a three-way node is clipped by the contact kernel into the correct five-point shape instead of collapsing into the wrong endpoint. For difficult three-way solid-wedge angles, the join uses extended contour intersections so the adjacent normal bond and the wedge share the same true intersection point. Tests were added or rewritten for hashed wedge retreat against connected single bonds, both-endpoint retreat, hash bond retreat, label clipping without extra retreat, hash obstacles being ignored by other contacts, center-double outer line cases, solid/dashed center-double outer lines, three-way/four-way contacts, and solid-wedge extended intersections.

Label anchors were the most iterated area today. The final rule is:

- Ordinary attached labels keep glyph anchor semantics. Right-side anchors prefer terminal letters and skip trailing numeric/subscript suffixes.
- CDXML centered labels have a dedicated centered layout. `LabelDisplay="Center"` or centered text justification imports as `attached-group-center`, `align=center`, and `anchor=middle`.
- Centered labels preserve bbox, box value, and glyph polygons, and export back with `LabelDisplay="Center"` / `LabelJustification="Center"`.
- Ordinary label hover, bond drag, and reopened text edit use glyph polygon centers. Centered labels use whole-text bbox center and the glyph box closest to the center.

Tests use `N3`, `Ph`, invalid `X3`, and centered numeric suffix labels from fixtures to cover these cases, including import -> export -> import stability.

### Selection Resize Handles

The last feature of the day was selection resize. The Rust engine added `selection_resize_drag` state and `EditorCommand::ResizeSelection`, with WASM methods:

```text
beginSelectionResize(handle, x, y)
updateSelectionResize(x, y)
finishSelectionResize(x, y)
```

Eight handles are supported:

```text
n, s, e, w, ne, nw, se, sw
```

Edge handles scale one axis, while corner handles scale proportionally. Resize drag state stores initial selection bounds, original selected node positions, original selected scene object payloads/transforms, undo state, and changed state. The undo snapshot is pushed only after real geometry change, so clicking a handle without moving does not dirty history.

The core scaling logic lives in `engine/select/drag.rs`:

- Scale is computed from the handle and opposite pivot.
- Minimum scale is clamped to `0.05` to avoid zero-size or flipped geometry.
- Molecule nodes are scaled in world space around the pivot, then written back to fragment-local coordinates.
- Attached node label geometry, fragment bounds, and symbol chemistry are refreshed after scaling.
- Text, bracket, symbol, shape, arrow, and related scene objects update transform translate, payload bbox, boxes, points, named points, text width/height, and graphic dimensions according to type.
- Circle/ellipse shapes that use absolute points are handled specially to avoid double-scaling transform and payload data.

The viewer computes eight handle positions from the current selection bounds, performs pointer hit testing, and uses matching resize cursors. Dragging calls the new engine resize API and displays a scale percentage overlay. CSS added `.editor-selection-resize-handle` and `.editor-selection-resize-label`. Tests cover east-handle one-axis scaling, object geometry scaling, and proportional corner scaling.

This makes the select tool much more complete: selected objects can now be moved, rotated, arranged, flipped, recolored, and resized, with core mutation still owned by the Rust engine.

### Follow-Up Notes

The desktop app now has a Tauri shell, native document service, hybrid native probe, native files/clipboard/export, multi-document tabs, and detached windows. The main editing loop still primarily uses the WASM host. When switching to the native engine path, continue migrating through `EngineHost` / `DesktopDocumentService` in stages rather than scattering direct Tauri command calls through the UI.

The custom titlebar and document tabbar now own much of the window behavior. If native menus are re-enabled later, define their relationship with the HTML command system clearly to avoid split sources for recent files, focused-window commands, and menu state.

The color picker is now a Chemcore-owned dialog. Follow-up work should cover focus restoration, keyboard accessibility, custom color persistence, and small-screen/high-DPI layout validation.

CDXML label anchors have converged on two separate rules for ordinary attached labels and centered labels. Future changes to the glyph kernel, label layout, or CDXML label import/export should run the related render, text, and bond tool tests first.

Selection resize is now in the engine, but more coverage is still needed for mixed multi-object selections, rotated transforms, curved arrows, absolute-point shapes, undo/redo, combinations with arrange/flip/rotate, and CDXML/SVG export stability after resize.
