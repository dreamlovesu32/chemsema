# Chemcore 开发者日志 - 2026-04-25

作者：张家骏

时间范围：2026-04-24 00:28 至 2026-04-25 00:47，Asia/Shanghai

## 总结

今天这轮工作的核心，不是继续往旧逻辑上补条件，而是把键接触绘制真正改造成统一的几何内核。此前主键、副键、粗实键、楔形键、虚键、中心双键之间的接触规则散落在不同分支里，很多地方靠局部启发式拼接，结果是前台容易出现毛刺、白缝、错误退让，以及预览和落地不一致。

今天把这一层重写成了以轮廓线和端点 profile 为中心的 Rust 渲染路径：实键统一看成多边形，接触统一按轮廓求交，拖动预览也直接走同一套 `render_document`。后面又把 hash 家族对 centered double 的边界规则收到了最终口径：`hash bond` 和 `hashed wedge` 都不再去斜切求交，而是保持标准母体形状，仅沿自身轴线退让。

## 键接触内核重写

今天最大的改动是重写了 fragment bond 的接触几何。

新的方向是：

- 对“有主键”的对象，不再分别写一堆局部 join 规则，而是统一抽象成端点轮廓；
- 两根键接触时，用各自内外侧轮廓线的延长线求交；
- 三根及以上键接触时，按节点周向顺序处理相邻 pair，每根键吃自己那一段端点轮廓链；
- `180°` 直通情况单独例外，不强行求不存在的稳定交点。

这一版的重点不只是“能算出来”，而是把过去那种 patch 盖 patch 的方式尽量撤掉，改成直接生成正确的 bond polygon。多键节点里一根键是五边形，已经不再是特殊补丁，而是端点 profile 自然推出来的结果。

旧的 legacy molblock 路径也从主渲染文件里分离到了独立文件，避免旧分支继续污染新的 contact kernel。

## 统一的键几何模型

今天把大量键型统一到了同一套几何口径下：

- 所有实键都按多边形处理，而不是 `line`。普通单键和粗实键只是宽度不同，几何本质相同。
- 实楔形键不再按三角形处理，而是按窄端很短的梯形处理，这样后续接触和宽度调整都能放到同一个多边形框架里。
- 虚键不再依赖 SVG 虚线。现在是先生成母体多边形，再用白色 knockout 等距切段。
- 哈希键和虚锲形键也统一成“粗实母体 + 更密的白缝切段”，其中黑色段长度保持等长，白色段允许在一定范围内波动，超过范围才增减段数。

这一点很重要，因为接触规则终于不再依赖“当前画的是 line 还是 polygon”。只要对象有明确轮廓，接触就都能回到同一套轮廓求交逻辑。

## 双键规则收敛

今天对双键做的工作，主要是把规则从“看上去差不多”收成“每一类都有明确判定”。

单侧双键现在的口径是：

- 外副线默认保持原样，不参与普通接触；
- 内副线才参与接触求交；
- terminal 端与主键等长，非 terminal 端按“与主键间距 * `sqrt(3) / 3`”缩短；
- 如果主键与相连主键构成锐角，而对方没有同侧副线，则副线退让；
- 如果对方也有同侧副线，则同侧副线仍然直接求交；
- 如果对方是中心双键，则用其中心线作为退让参考。

中心双键也重写了：

- 不再把它当成“特殊双线装饰”，而是两根独立子线各自判断各自的端点；
- 每端、每侧单独决定是否延长到其他键的轮廓；
- 大于 `162°` 的近直通情况保持原样，不再强行延长；
- 混合实虚中心双键、双虚中心双键也都纳入了同一套 centered-double 规则。

这让 side double 和 centered double 的行为边界终于清楚了，也让等长双键和分叉键接触时能按轮廓真正接上。

## 哈希键与多键节点

哈希键和虚锲形键今天单独定了两条关键规则。

第一条是它们和普通主键接触时，母体外框保持标准形状不变：

- `hash bond` 始终保持标准矩形；
- `hashed wedge` 始终保持标准梯形。

它们不再像实键那样主动变形去贴合普通键。今天后半段又把 centered double 这条特例收到了最终规则：

- `hash bond` 面对 centered double 时，矩形端帽不求交，只沿自身轴线退让；
- `hashed wedge` 也不再宽端求交，而是保持标准梯形，只沿自身轴线退让；
- 虚锲形键因为宽端宽度固定，长度变化后梯形会自然变形，但这仍然是“标准梯形缩短”，不是改成一块异形接触补丁；
- 两者的黑色段长度继续保持一致，白色段允许在范围内浮动。

第二条是多键节点中的优先级：

- 如果一个端点同时接多根键，哈希键和虚锲形键保持原样；
- 其他键退让，并且故意留出一点白，不做完全无缝贴合。

这避免了 hash 家族在复杂节点里被强行挤压到失真，也让视觉层级更稳定。

## 编辑器预览与前台清理

今天除了后端几何，前台也做了几件很关键的收口工作。

首先，拖动画键时的预览不再是一根假的 overlay 线。现在 Rust engine 会克隆当前文档，临时插入待画键，再直接跑一次 `render_document`。也就是说：

- 拖动时看到的预览；
- 松手后真正写入文档的结果；
- 渲染内核最终输出的 display list；

三者现在用的是同一套几何。

其次，调试阶段临时加的右下角坐标和顶点面板已经移除，没有把这套 UI 残留在正式编辑器里。左侧主 bond 工具图标也改成会跟随当前二级键型同步。

另外，前台对 `strokeWidth: 0` 的处理、polygon 是否描边、以及共享边的样式覆盖也做了清理。之前一些“后端 SVG 正常、前台 viewer 有毛刺/白线”的问题，最终都定位成前台把无描边 polygon 又错误地描了一圈。

## 验证

今天主要使用的验证命令是：

```bash
cargo test
npm run build:engine-wasm
node --check viewer/app.js
```

其中 `cargo test` 覆盖了：

- bond tool 交互测试；
- render document 几何回归；
- 多键节点、中心双键、哈希键、楔形键等接触场景。

另外，这轮还在前台实际交互里逐条验收了：

- 两键接触；
- 三键和多键接触；
- 单侧双键与中心双键规则；
- 哈希键、虚锲形键与普通键/中心双键/多键节点的边界行为；
- 哈希键与虚锲形键对中心双键的“退让但不改母体形状”最终规则；
- 虚锲形键白缝跟随实际梯形，而不是按旧长度切段；
- 拖动画键预览与松手落地的一致性。

## 风险和下一步

今天这轮最大的收获是，键接触终于从“修图式前台补丁”变成了“后端几何定义”。但仍有几件事后面要继续盯：

- centered double、side double、hash 家族虽然已经进了统一框架，但规则很多，后续还要继续补更细的例子，避免边角回归。
- 当前前台和 Rust display list 已经基本一致，下一步应该继续减少 viewer 里对几何细节的二次解释。
- 接触规则已经大幅收敛，后续更适合把它们整理成独立文档，而不是只散在测试名和代码分支里。

今天的架构性结论是：键绘制真正稳定下来，不靠 viewer 里继续加条件，而是要把“键是什么几何、端点怎么闭合、节点怎么分配接触区”都收回 Rust 内核里统一定义。
