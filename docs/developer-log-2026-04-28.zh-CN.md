# Chemcore 开发者日志 - 2026-04-28

作者：张家骏

时间范围：2026-04-28 00:00 至 2026-04-28 23:59，Asia/Shanghai

维护说明：这份日志最初记录的是当日从像素模型向物理单位模型收口的过程。当时使用了 `cm` 作为过渡说法；当前项目规则已经在 2026-04-30 收敛为 `format.unit = "pt"`。下面的单位表述已按当前 `pt` 口径更新。

## 总结

今天这轮不是单点修 bug，而是把编辑器里一整串“单位、几何、编辑态、选择、删除、渲染边界”问题一起往正确方向收了一遍。真正的主线有五条：

- 内部几何继续脱离裸 `px`，并把一批历史像素常量、渲染偏移、默认值往统一点数尺度和 typed unit 边界收；
- 默认样式与交互几何按 ChemDraw 风格重设，包括默认键长、键宽、文本字号、标签字号、聚焦框、聚焦圆、粗实键/哈希键宽度、楔形键宽度等；
- 文本编辑态从浏览器 DOM 几何继续往内核迁移，解决字体缩放、黑框错位、点击位置和可见文字不一致、进入编辑态轻微拉伸或下移等一整类问题；
- `engine.rs` 里过大的文本编辑、删除、选择逻辑开始拆分，Viewer 继续退回到“输入事件 + 世界坐标换算 + primitive 渲染”的角色；
- 选择和删除链路正式接上，并把键选中框改成按整根键的实际渲染轮廓算包围盒，不再只包主键中心线。

这一轮最重要的结果，不是某一个界面细节终于“看起来对了”，而是几何真相继续从浏览器测量和前端补偿手里收回到了内核和渲染共用逻辑里。

## 单位体系继续收口到 pt

今天一开始最明确的决策，是继续把内部几何从浏览器像素里抽离出来。当前维护口径下，文档和 engine 世界坐标统一按 `pt` 解释，避免“内核拿一种单位，前端拿 px，再来回猜”的状态。

### 为什么继续选 pt

原因并不抽象：

- ChemDraw/CDXML 坐标本来就更接近页面点数坐标；
- 未来要多端同步，桌面端和浏览器端如果内部几何用的不是同一套单位，回放和编辑态会越来越难对齐；
- 键长、线宽、字号、双键偏移、聚焦框这类参数，本质上都更适合用文档点数，而不是屏幕像素。

今天的方向不是把所有 px 直接删掉，而是把 px 限制在 viewer 边界和输入命中半径这种确实和 CSS 设备像素强相关的地方。

### typed units 与边界转换

这轮还继续往更稳的方向迈了一步：开始显式区分 `CssPx` 和 engine 世界坐标这类量纲边界。  
长期目标不是靠约定记忆“这里应该传文档单位”，而是让边界函数显式做转换，让 `px` 混进内核几何时更早暴露。

这件事已经开始体现在常量层和部分渲染层：

- `render_constants.rs` 里已有一批 typed 常量；
- 一部分旧的像素换算高频常量调用已经改为 typed 常量源引用；
- 后续 viewer 和 engine 之间的边界函数会继续显式标出单位转换，而不是到处散落裸 `f64`。

## 默认样式与几何参数重设

今天按用户确认过的参数，把默认样式进一步往 ChemDraw 靠拢。

### 默认值

当前认可并落实的核心默认值是：

- 默认键长：`30 pt`
- 默认键线宽：`1 pt`
- 默认文本字号：`7.5 pt`
- 默认原子标签字号：`7.5 pt`
- 默认字体：`Arial`

这组值不只是工具栏显示值，而是继续往文档默认样式、文本编辑默认 session、标签默认生成规则里统一。

### 双键与多键参数

今天对双键、三键以及相关宽度规则的整理，主要有这些：

- 双键偏移仍以键长的 `0.12` 为基线；
- 但这个距离重新明确为“两根线内侧的距离”而不是中心线距离；
- 所以双键真实中心距要把线宽一起算进去；
- 粗双键时，如果主键加粗，两根线间距也必须跟着重新计算，不能保持不变；
- 三键延续同一偏移基线；
- 粗实键和哈希键默认宽度统一为 `4 pt`；
- 楔形键的宽端宽度是粗实键宽度的 `1.5` 倍；
- 楔形键的窄端宽度回到默认键宽，也就是 `1 pt`。

这些调整的目标很一致：多键和宽键的视觉比例要和默认键宽、默认键长一起联动，而不是各自藏一套独立倍率。

### 聚焦与 hover 几何

今天还明确收过一批聚焦尺寸：

- 聚焦框宽度：约 `5.67 pt`
- 聚焦框长度：约 `22.68 pt`
- 端点聚焦圆半径：约 `2.83 pt`

这里做的不只是数值替换，而是把它们从“看着差不多”的像素值，继续拉回到和整体点数几何一致的尺寸体系里。

## DPI、浏览器缩放与 viewer 边界

用户今天还明确提出了一个很关键的问题：DPI 不能被写死。浏览器端应该尽量尊重用户当前设备和系统缩放，比如 150% 缩放下等效接近 `144 dpi`。

### 决策

这里没有去把“144 dpi”硬编码成新的固定值，而是明确了边界：

- 内核几何仍然只认文档点数；
- viewer 负责读浏览器环境中的实际缩放信息；
- `pt -> CSS px` 的换算应当依赖当前环境，而不是假设固定 `96 dpi`。

这样做的原因很直接：

- 未来多端同步时，文档几何不能跟某一台机器的 DPI 绑定；
- 但浏览器显示又不能完全忽略用户的系统缩放和 `devicePixelRatio`；
- 所以内核和文档存储层必须稳定，显示层再根据当前设备做边界换算。

今天相关测试里也保留了 `DEVICE_SCALE_FACTOR=1.5` 的回放，以便模拟用户提到的 150% 缩放环境。

## 常量源与渲染侧常量收口

用户提出了一个很对的建议：把常量收成集中来源，并尽量带上引用点备注，避免以后又漏改一块。

今天沿着这个方向做了两件事：

- 继续把渲染侧高频的旧像素换算常量往 `render_constants.rs` 和 typed 常量源收；
- 明确后续优先清理 `render.rs` / `render_bonds.rs` 里那批楔形键、虚线、箭头、label clip margin、bond offset 相关常量。

虽然今天不可能一次把所有旧常量全部消掉，但方向已经很明确：  
高频几何常量应该有单一来源，viewer 和 engine 都不要各抄一份“差不多的数字”。

## 文本编辑器继续内核化

今天花得最多的时间，仍然是文本编辑态。原因也很明确：前面 unit 迁移后，文本编辑暴露出一整串浏览器和内核模型混杂的问题。

### 暴露出的症状

今天实际追过的问题包括：

- 进入编辑态后文字轻微拉伸；
- 进入编辑态后文字会轻微向下位移；
- 点击进入编辑时，文字显示成 `9.8`、`9.999` 这种不该出现的数字漂移；
- 黑框包不住文字；
- 黑框改对后，文字又重新缩放；
- 点击位置、可见 I 光标、实际文本框起点三者不一致；
- 打字过程中甚至出现“看不到字”或编辑层整体跑偏。

用户指出的核心判断是对的：  
如果前端只是显示层，那么这些问题不应该靠 viewer 一直补坐标，而应该把文本几何重新收到内核里。

### 从 DOM 几何回到内核 layout

这轮最终采取的不是继续修 `getBBox()`、`renderOffset`、`DOM measuredSize` 这些壳层，而是把编辑态几何继续往内核推进：

- 编辑态文字的 line layout、anchor、bbox、caret、selection rect 继续统一由内核产出；
- viewer 的 `textarea` 只负责输入与 IME；
- 可见层改成 SVG `text/tspan` 渲染；
- 黑框、光标、选择区都跟随内核 layout 结果，而不是再让 DOM 反推几何。

这使得“显示态是什么样、编辑态就尽量还是什么样”开始成为可实现目标，而不是一条不断被浏览器排版打断的期望。

### 编辑锚点与文本框起点

今天还专门处理了文本框进入编辑态时的锚点语义。

用户给出的目标很明确：  
文本框更合理的起点应该是 I 光标左上角，而不是 I 光标中部附近的某个补偿点。

这轮因此继续把编辑态锚点朝下面这套语义收：

- 点击点决定逻辑起点；
- 编辑框 origin 更接近 I 光标左上角；
- 文本显示层与提交后的文字位置保持一致；
- 重开编辑时优先回到稳定锚点，而不是根据浏览器测量重新飘。

### `measuredSize` 退出协议边界

另一件结构性变化是：`measuredSize` 这类历史 DOM 尺寸字段继续退出边界。  
Viewer 不再把浏览器量出来的文字块尺寸当成“可以提交回内核的几何真相”。

这一步很关键，因为只要 DOM 测量结果还能回流进内核，点数体系就永远会被一套浏览器私有排版逻辑污染。

## 文本工具栏与默认字体

今天还顺手把文本工具栏行为对齐了一轮：

- 字号输入不再允许长期停留在 `9.999` 这种显示上明显错误的状态；
- 字号改成带下拉列表的选择形式；
- 默认字号回到 `10`；
- 默认字体明确改成 `Arial`；
- 在用户还没输入任何内容时，默认字体和字号就应当已经是这组值，而不是等输入后再被动修正。

这部分看起来偏 UI，但其实也是“默认 session 值必须和文档/内核默认值一致”的一部分。

## Bond 工具、标签聚焦与方向规则

今天还继续收了一些 bond-tool 和标签工具的规则，这些内容虽然零散，但都和整体编辑体验密切相关。

### 标签在 bond 模式下的字符级聚焦

在文本模式下，普通文本框仍按整块聚焦；  
但在 bond 模式下：

- 普通文本框不参与聚焦；
- attached label 按字符级聚焦；
- 每个字符都可以作为拖出键的起点；
- 垂直上下拖时，允许保留当前字符中心做锚点；
- 左向拖时吸到最左字符；
- 右向拖时通常吸到从右往左第一个大写字母；
- 但如果用户本来就是从最右字符直接往右拖，则保留最右字符，不强制回退到大写字母组。

这套规则已经下沉到内核测试里，而不是靠 viewer 侧命中猜测。

### 画键后 hover 不应跳端点

用户还指出一个交互问题：画完键以后，hover 会自动跳到新端点，而不是停留在鼠标指的位置。  
今天修正后，画键成功收尾不再强塞新端点 hover，而是按当前 pointer 位置重新做 bond-mode hit test。

这让“鼠标在哪，聚焦就按哪算”重新成立。

### 双键默认方向

今天还明确了一条更细的双键默认规则：

- 只有在同一边已经有两个或以上取代基，另一边没有取代时，第一次画出来才默认是等长双键；
- 其余情况默认应该走单侧双键；
- 方向判断本身应继续由前面的结构逻辑决定，而不是做成纯显示层切换。

## 删除模式正式接上

删除这条交互链今天也终于成型了。

### 删除工具

左侧/顶部的删除入口最终统一成进入删除模式的工具，而不是又多出一套平行图标逻辑。  
删除模式下：

- 光标进入删除态；
- 点击单键删除整根键；
- 点击双键先降级成单键；
- 点击三键先降级成单侧双键；
- 点击端点删除这个点连接的所有键；
- 点击标签先删标签本身，端点保留；
- 点击普通文本框删除文本对象。

### 非删除模式下的 Delete 键

在其他模式下按 `Delete`：

- 如果当前聚焦在键上，直接删除整根键，不管它是单键、双键还是三键；
- 聚焦在端点、标签、文本框上，则分别走对应对象的删除逻辑。

## 选择模式正式接上

今天选择模式不只是“可以框选”，而是把选择对象、显示规则、分子内部聚合逻辑都定了下来。

### 选择方式

当前支持：

- 点选；
- 矩形选；
- 套索选；
- `Shift` 追加多选。

### 分子内部的显示规则

用户最后确认过的规则是：

- 同一分子内如果只选中一个对象，就显示这个对象自己的框和中心蓝点；
- 如果同一分子内选中多个对象，就不再重复画每个小框，只画包住所有小框的最小大框；
- 内部的键、原子、标签都只保留中心蓝点；
- 不同分子各自算自己的最小框。

对象级小框的定义也明确了：

- 原子：使用端点方框；
- 标签：使用标签文本框；
- 键：使用刚好包住整根键轮廓的轴对齐矩形。

### 键选中框改成真实渲染轮廓

今天最后又修了一次用户指出的细节：  
“包住整根键”不只是单键主键中心线，而是双键三键也都要包进去。

因此最终没有继续在 `select.rs` 里写更多偏移特判，而是直接改成：

- 在 `render.rs` 里新增 `fragment_bond_visual_bounds(...)`；
- 复用现有 bond 渲染逻辑，为单根键生成实际 `DocumentBond` primitives；
- 对 line / polygon / polyline 求真实包围盒；
- `select.rs` 只消费这个结果。

这样一来：

- 双键选中框会包住两根线；
- 三键选中框会包住三根线；
- 粗键和楔形键也会自然被包住；
- 以后只要渲染参数变了，选中框会自动跟着渲染外轮廓走。

## `engine.rs` 拆分与 Viewer 继续瘦身

今天做的另一件对维护性很重要的事，是开始拆 `engine.rs`。

### Rust 侧拆分

这轮拆出的模块包括：

- `crates/chemcore-engine/src/engine/text_edit.rs`
- `crates/chemcore-engine/src/engine/delete.rs`
- `crates/chemcore-engine/src/engine/select.rs`

收益很直接：

- 文本编辑、删除、选择三条复杂交互链不再互相缠在一个几千行主文件里；
- 后面继续查问题时，范围能直接缩小到模块级。

### Viewer 继续退回显示层

Viewer 这边今天的总体方向仍然没变：

- 处理键盘/IME 输入；
- 把屏幕坐标换成世界坐标；
- 消费 wasm 暴露出来的状态与 primitive；
- 渲染 SVG / HTML 层。

旧的浏览器几何壳层继续被清掉，尤其是那些会重新定义文本框尺寸、标签锚点或 bond 几何的逻辑。

其中覆盖的重点有：

- 删除工具和 `Delete` 键两条链路；
- 字符级标签聚焦与拖键；
- 单键、双键、三键、粗键的选择框回归；

此外，编辑态相关测试还保留了高缩放环境回放，用来模拟 150% 缩放这类用户实际场景。

## 风险和下一步

今天这轮虽然已经把很多问题从 viewer 壳层里收回来了，但后面仍有几件事值得继续做：

- 把剩余高频旧像素换算常量继续往 typed 常量源收，尤其是楔形键、哈希键、箭头、label clip margin 这批渲染接缝参数；
- 文本编辑虽然已经基本不再依赖 DOM 几何，但 caret、selection、IME 边界仍可继续往内核结果驱动收紧；
- 选择框已经开始走真实渲染包围盒，后续如果要处理更复杂对象，也应继续坚持“渲染即真相”，不要再回到手写近似框；
- viewer 和 engine 之间的单位边界还可以继续显式化，尽量让 `CssPx -> 文档点数` 转换只发生在少数明确入口。

今天的结论可以概括成一句话：  
这一天不是在前端补更多坐标，而是在把单位、几何、文本编辑和选择删除这些长期容易漂移的东西，继续拉回到一套更稳定的内核和渲染共用模型里。
