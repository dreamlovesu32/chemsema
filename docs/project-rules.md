# Chemcore Project Rules

This document records project-level rules that must stay true during the current development phase. More detailed behavioral rules live in topic-specific documents such as format, bond rendering, and command history.

The long-term plan for the Windows desktop app and Office integration is described in `docs/windows-desktop-office-architecture.md`. Its core constraint is that desktop and Office integration layers must not duplicate chemical logic; they must continue to work through the Rust engine and the unified document service.

## Engine Boundary

- Rust `crates/chemcore-engine` is the authority for current editing behavior, document mutation, hit testing, snapping, selection, deletion, command history, and render primitives.
- The viewer is responsible only for toolbar, menus, file open/save, browser event collection, coordinate conversion, and SVG/DOM rendering.
- New chemical editing behavior should not be scattered back into `viewer/app.js`. If the viewer needs geometry, it should consume engine primitives or explicit engine state.
- WASM is the runtime form of the same Rust engine for browser and desktop hot editing paths; business rules still live in the Rust engine.
- The Windows desktop app defaults to `DesktopHybridEngineHost`: high-frequency behavior such as pointer move, hover, focus, hit testing, selection, drag preview, rotation, scaling, movement, and object settings must remain in-process core calls.
- The Tauri native service owns system capabilities such as files, clipboard, export, Office/OLE, windows, menus, recent files, and background tasks.
- High-frequency mouse events must not be designed as synchronous cross-Tauri IPC calls that return a full document/render/state JSON snapshot every time. Unless incremental diffs, event coalescing, and large-file performance are proven, `TauriEngineHost` is only a `?engine=tauri-native` diagnostic path.
- Context menus, scale/rotation panels, object settings, and similar UI may be displayed by the viewer, but field definitions, applicable objects, mixed values, apply logic, and document mutations must come from engine APIs.

## Office/OLE Boundary

- Office integration must be bounded by the independent COM/OLE local server in `apps/chemcore-office`; the OLE lifecycle should not be embedded directly in the desktop main-window process.
- Chemcore's OLE class is fixed as `Chemcore.Document` / `Chemcore.Document.1` / `{CB69F54F-F21E-44DE-84FB-89D98FECE056}`.
- Development registration writes to `HKCU\Software\Classes`; the production installer writes to `HKLM\Software\Classes`. Users should not be asked to edit the registry manually.
- The OLE server is responsible only for COM/OLE interfaces, storage, preview, clipboard objects, and waking the desktop app. Chemical parsing, document mutation, import/export, and rendering semantics remain provided by the Rust engine / desktop service.
- Office Add-in support may later enhance Ribbon or entry points, but it cannot replace the OLE embedded object.

## Document Units

- The document JSON unit in current `.ccjs` / `.ccjz` native files is fixed as printing points: `format.unit = "pt"`.
- Document coordinates, object positions, bond length, line width, font size, hit radius, paste offsets, and other persisted or engine world-coordinate values are interpreted in `pt`.
- CSS pixels may appear only at the viewer boundary and browser input/display adaptation layer. They must be explicitly converted before entering the engine.
- Historical names such as `WorldCm` or `*_CM` may still exist in code; their current semantics should be understood under the `pt` rule. Renaming them later must be an isolated refactor, not mixed into behavior changes.
- `cm` rules that appear in old documents or logs were superseded by the `pt` decision made on 2026-04-30.

## WASM Synchronization

- During routine development, Rust source and generated `viewer/engine` artifacts may be briefly out of sync.
- Before validating engine behavior in the viewer, rebuild the Web engine:

```powershell
npm run build:engine-wasm
```

- For frequent Rust engine edits, it is recommended to run an auto-rebuild process:

```powershell
npm run dev:engine
```

- Before committing, delivering, or asking others to validate the viewer, run:

```powershell
npm run verify
```

This command runs Rust tests, rebuilds engine WASM, checks viewer syntax, and confirms that `viewer/engine` has no uncommitted generated artifact differences.

## Generated Artifacts

- `viewer/engine/chemcore_engine.js`, `viewer/engine/chemcore_engine.d.ts`, and `viewer/engine/chemcore_engine_bg.wasm` are runtime generated artifacts for the Web viewer.
- After changing `crates/chemcore-engine/src/wasm.rs`, engine APIs, or render primitive structures, these generated artifacts must be synchronized.
- The `viewer/engine/.gitignore` generated by `wasm-pack` should not be kept; the build script removes it.

## Rendering Geometry

- Fragment-path bond rendering rules use `docs/bond-rendering-rules.md` as the behavioral baseline.
- Bond contact, label clipping, dash/hash knockout, preview-state geometry, and committed-state geometry should be uniformly defined by the Rust render path.
- The viewer should not redefine chemical bond geometry through SVG `linecap`, extra center patches, or foreground fixes.

## Chemical Labels

- Implicit hydrogen behavior uses `docs/implicit-hydrogen-rules.md` as the current behavioral baseline.
- Abbreviation recognition and functional-group expansion use `docs/abbreviation-recognition-rules.md` as the current behavioral baseline.
- Formula-like label valence parsing uses `docs/valence-label-recognition-rules.md` as the current behavioral baseline.
- The semantics of charges, radicals, and lone-pair symbols after they are assigned to molecular atoms use `docs/charge-radical-symbol-rules.md` as the rules baseline.
- Label recognition, implicit hydrogen counts, generated label text, and bond-drawing anchors are Rust engine behavior and should not be reimplemented in the viewer.
- CDXML import is an input adapter, not a separate label-layout engine. It may
  read `t` positions, bounding boxes, runs, alignment, and script information
  from CDXML, but it must translate them into ChemCore's native node-label
  model. Label anchors, display ordering, glyph polygons, and bond retreat stay
  owned by the Rust engine.
- `meta.import.cdxml` is provenance and round-trip/debug metadata for data that
  actually came from CDXML. Source-neutral measured label geometry from OCR,
  screenshots, pasted images, or future non-CDXML inputs must not be encoded as
  `import.cdxml` or rely on CDXML-import anchor compatibility paths. Use a
  source-neutral measured-geometry contract instead.

## Text Symbols And Glyphs

- Unicode symbols and special characters in text use `docs/text-symbol-glyph-profile-rules.md` as the current behavioral baseline.
- `shared/text_symbols.json` only defines UI grouping for the text-symbol palette; after a character enters a document, it is ordinary text run content.
- Glyph advance, ink box, background box, glyph polygon, and unknown-character fallback are uniformly defined by the Rust glyph kernel. The viewer may only consume shared profiles and engine layout.

## Command History

- Only committed document changes enter history.
- Hover, focus halos, preview, lasso, active drag, and caret movement are transient interaction state and do not enter history.
- New editing features should use semantic `EditorCommand`; `legacy-mutation` should be treated only as a migration warning.
